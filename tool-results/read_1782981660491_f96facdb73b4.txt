    44→- Known environment quirk: background processes spawned with `&`/`nohup`/`setsid` (without `--fork`) are killed by the sandbox at the end of each shell tool call (they remain in the call's process group and die with the `su z -c bash` wrapper, PID 7199-style).
    45→
    46→**FIXED (round 2): persistent dev server via `setsid --fork`.**
    47→- Root cause: each Bash tool call runs as `su z -c '/bin/bash ...'` (a child of the python agent, PID ~7199). When the call ends, that shell and its whole process group/session are killed. `agent-browser` survives because it is a properly daemonized orphan (PPID=1).
    48→- Solution: launch the dev server (and a watchdog) with `setsid --fork` so they fork into a new session and become orphans reparented to PID 1 (tini). Verified: a `setsid --fork sleep 300` survived across separate Bash calls; the dev server (PPID=1) returned HTTP 200 in a subsequent call.
    49→- Created `/home/z/my-project/keepalive.sh` — a watchdog loop that curls localhost:3000 every 10s and restarts the server (via `setsid --fork`) if it is down. The watchdog itself is launched with `setsid --fork` so it persists (PPID=1).
    50→- Launch commands (re-runnable if everything dies):
    51→  ```bash
    52→  cd /home/z/my-project
    53→  setsid --fork bash -c 'cd /home/z/my-project && exec bun node_modules/next/dist/bin/next dev -p 3000' </dev/null >>dev.log 2>&1
    54→  setsid --fork bash /home/z/my-project/keepalive.sh </dev/null >>keepalive.log 2>&1
    55→  ```
